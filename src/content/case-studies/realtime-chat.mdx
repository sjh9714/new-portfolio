## 요약

Realtime Chat은 WebSocket 연결 자체보다 다중 인스턴스 환경에서 권한, 순서, presence, reconnect 복구가 어떻게 흔들리는지에 초점을 둔 채팅 백엔드 사례입니다. STOMP subscribe authorization, Kafka ACK/NACK, roomId key ordering, Redis presence, reconnect sync API, DLT replay를 각각 분리해 설계했습니다.

채팅방 조회 API는 937 RPS에서 1,598 RPS로 개선했고, p95 응답 시간은 212.85ms에서 149.22ms로 줄였습니다. N+1 쿼리는 2N+1회에서 1회로 제거했습니다. WebSocket 연결 스모크 테스트는 2대 합산 1,158 sessions, 연결 체크 성공률 100%를 기록했습니다.

## 문제

단일 인스턴스 채팅은 WebSocket 연결과 메모리 상태만으로도 어느 정도 동작합니다. 하지만 인스턴스가 둘 이상이 되면 client A가 App #1에 붙고 client B가 App #2에 붙는 순간부터 문제가 바뀝니다. 구독 권한은 연결 시점이 아니라 subscribe 시점에도 검증해야 하고, 메시지는 같은 방 안에서 순서가 흔들리지 않아야 하며, presence는 프로세스 메모리에만 둘 수 없습니다.

또한 네트워크가 끊겼다가 reconnect되는 사용자는 마지막으로 어디까지 받았는지 기준이 필요합니다. “연결이 살아 있었다”와 “모든 메시지를 받았다”는 다른 주장입니다. 그래서 이 프로젝트는 연결 smoke, API 성능 수치, 50-user repeat3와 500-user repeat3 receiver matrix local scenario를 분리하고, 1,000 session benchmark와 운영 성능 주장은 별도 추가 측정 예정 근거로 남겼습니다.

## 단순 구현에서의 문제

가장 단순한 구현은 WebSocket session registry를 한 인스턴스 메모리에 두고, 메시지를 그 registry에 바로 fan-out하는 방식입니다. 이 방식은 로컬에서는 빠르게 보이지만 인스턴스가 늘면 다른 서버에 붙은 구독자를 찾지 못합니다. subscribe authorization을 생략하면 방 멤버가 아닌 사용자가 topic을 직접 구독할 수도 있습니다.

Kafka를 붙여도 roomId key를 고려하지 않으면 같은 방 메시지 순서를 설명하기 어렵습니다. Redis presence도 TTL과 heartbeat 기준이 없으면 “온라인처럼 보이지만 실제로는 끊긴 사용자”가 남습니다. 그래서 각 역할을 단순 연결이 아니라 실패 가능한 경계로 나눴습니다.

## 설계 판단

STOMP subscribe authorization은 사용자가 특정 room topic을 구독할 권한이 있는지 확인합니다. 메시지 publish는 Kafka producer ACK/NACK 결과를 기준으로 성공과 실패를 분리하고, roomId를 key로 사용해 같은 방의 순서를 하나의 partition 안에서 유지하도록 설계했습니다.

Redis presence는 최종 기준 데이터가 아니라 짧게 살아 있는 접속 상태입니다. 끊김 이후 복구는 Redis presence가 아니라 Message DB와 reconnect sync API가 담당합니다. client가 마지막으로 받은 위치를 보내면, 서버는 DB 기준으로 이후 메시지를 다시 조회합니다.

읽음 상태와 메시지 목록 조회는 N+1이 발생하기 쉬운 영역입니다. 이 부분은 조회 API 성능 개선으로 먼저 측정했고, 실시간 전달 경로의 지연과 완전성은 별도 benchmark로 분리했습니다. 이렇게 나누면 “조회 API는 측정됨”과 “실시간 전달 품질은 아직 측정 전”을 혼동하지 않습니다.

## 검증 결과

측정 완료 근거는 채팅방 조회 API RPS 937 -> 1,598, p95 212.85ms -> 149.22ms, N+1 2N+1 queries -> 1 query입니다. 시나리오 검증 근거는 WebSocket 연결 스모크 테스트 1,158 sessions / 100% success, 50-user receiver matrix repeat3, 500-user receiver matrix repeat3입니다.

50-user receiver matrix repeat3는 세 번 모두 expected 4,900 / unique 4,900 / missing 0 / duplicate 0 / completeness 100%를 기록했고, p95 범위는 23-38ms입니다. 500-user receiver matrix repeat3도 세 번 모두 expected 49,900 / unique 49,900 / missing 0 / duplicate 0 / completeness 100%를 기록했고, p95 범위는 37-47ms, p99 범위는 46-233ms입니다. 10 rooms x 50 users mixed traffic local scenario는 세 번 모두 unique 4,900/4,900, missing 0, duplicate 0, receiver p95 18-20ms, mixed HTTP failed 0/30/run을 기록했습니다. 다만 모두 local single app / Docker Compose 실행이므로 production multi-instance benchmark나 운영 성능 주장으로 쓰지 않습니다. 추가 측정 예정 근거는 운영 조건의 send-to-receive p50/p95/p99와 delivery completeness입니다.

## 한계와 다음 검증

연결 smoke는 “연결 가능”을 확인할 뿐 “모든 메시지 전달”을 보장하지 않습니다. 다음 측정은 sender client SEND timestamp와 receiver client receive timestamp를 기준으로 latency histogram을 만들고, consumer failure 후 DLT replay로 누락 메시지가 복구되는지를 별도 시나리오에서 확인하는 방식이 적절합니다.
