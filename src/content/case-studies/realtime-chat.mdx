## TL;DR

WebSocket 채팅을 단일 인스턴스 기능 구현으로 끝내지 않고, STOMP 구독 인가, Kafka ACK/NACK, roomId 기반 ordering, Redis presence, reconnect sync, DLT replay 관점으로 다중 인스턴스 정합성을 검증했습니다.

## Problem

- 구독 권한을 느슨하게 두면 room 외부 사용자가 메시지를 받을 수 있습니다.
- Kafka publish와 consumer 처리 실패는 사용자에게 보이지 않는 메시지 유실로 이어질 수 있습니다.
- 다중 인스턴스에서 presence와 읽음 상태는 메모리만으로 유지하기 어렵습니다.
- 재연결 이후 누락 메시지 동기화 기준이 없으면 메시지 전달 신뢰도를 설명할 수 없습니다.

## Design Decisions

- STOMP subscribe 단계에서 room membership을 검증해 구독 자체를 권한 경계로 취급했습니다.
- roomId를 Kafka key로 사용해 같은 방 메시지의 순서 역전을 줄였습니다.
- Redis presence와 reconnect sync API를 분리해 연결 상태와 메시지 복구 책임을 나눴습니다.
- 아직 send-to-receive latency와 delivery completeness는 Pending으로 남겨 실제 지연/전달률 수치를 과장하지 않습니다.
